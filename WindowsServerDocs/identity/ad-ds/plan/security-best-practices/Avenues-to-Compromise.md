# <a name="avenues-to-compromise"></a>Esposizione a possibili violazioni

>Si applica a: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

*Legge numero sette: rete più sicura è una amministrata correttamente.* - [10 Immutable Laws of Security Administration](https://technet.microsoft.com/library/cc722488.aspx)  
  
Nelle organizzazioni che si sono verificati errori irreversibili compromissione eventi, le valutazioni rivela in genere che le organizzazioni sono limitate visibilità lo stato effettivo di infrastrutture IT, che potrebbero essere diverse in modo significativo dai loro Stati "come documentato". Tali variazioni introducono vulnerabilità che espongono un ambiente di compromissione dell'installazione, spesso con individuazione fino a quando non la compromissione è stata distribuita nel punto in cui gli utenti malintenzionati in modo efficace "proprietario" dell'ambiente di rischio.  
  
Valutazioni dettagliate di configurazione di Active Directory di queste organizzazioni, infrastrutture a chiave pubblica (PKI), server, workstation, applicazioni, access control list (ACL) e altre tecnologie di visualizzare gli errori di configurazione e le vulnerabilità che, se il problema è risolto, potrebbero aver impedito la compromissione iniziale.  
  
Analisi della documentazione, processi e procedure IT identifica vulnerabilità introdotte da eventuali lacune nei procedure amministrative che sono stati utilizzati da utenti malintenzionati per ottenere privilegi che sono stati usati per compromettere completamente l'insieme di strutture Active Directory. Un insieme di strutture completamente compromesso è uno in cui gli utenti malintenzionati compromettono non solo singoli sistemi, applicazioni o gli account utente, ma inoltrare l'accesso per ottenere un livello di privilegi in cui puoi modificare o eliminare tutti gli aspetti dell'insieme di strutture. Quando un'installazione di Active Directory è stata compromessa per che gradi, gli utenti malintenzionati possono apportare modifiche che consentono loro di mantenere una presenza in tutto l'ambiente o, peggio, per eliminare la directory e i sistemi e gestisce account.  
  
Anche se un numero di vulnerabilità sfruttata comunemente le descrizioni seguenti non è gli attacchi contro Active Directory, consentono di stabilire un mercato in un ambiente che può essere utilizzato per l'esecuzione con privilegi (detto anche elevazione dei privilegi) attacchi e per alla fine di destinazione e di dominio Active Directory di compromettere gli autori di attacchi.  
  
In questa sezione di questo documento si concentra sulla descrizione i meccanismi che gli utenti malintenzionati usano in genere per ottenere l'accesso all'infrastruttura e alla fine per avviare attacchi di elevazione di privilegi. Vedere anche le sezioni seguenti:  
  
-   [Ridurre la superficie di attacco Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md) dettagliate consigli per la configurazione di protezione di Active Directory.  
  
-   [Monitoraggio di Active Directory per segni di compromissione](../../../ad-ds/plan/security-best-practices/Monitoring-Active-Directory-for-Signs-of-Compromise.md) consigli per il rilevamento di compromissione  
  
-   [Pianificazione del compromesso](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Planning-for-Compromise.md) approcci di alto livello per la preparazione di attacchi contro l'infrastruttura da IT e prospettive aziendali  
  
> [!NOTE]  
> Sebbene questo documento è incentrato sui sistemi Windows e di Active Directory che fanno parte di un dominio di Active Directory, gli utenti malintenzionati concentrarsi raramente esclusivamente in Active Directory e Windows. Negli ambienti con una combinazione di sistemi operativi, directory, applicazioni e archivi dati, è comune per trovare che sono stati compromessi anche i sistemi non Windows. Ciò è particolarmente vero se i sistemi forniscono "ponte" tra ambienti Windows e non Windows, ad esempio i file server accessibili da Windows e UNIX o Linux client, le directory che forniscono servizi di autenticazione a più sistemi operativi o metadirectory che sincronizzare i dati in directory diverse.  
>   
> Servizi di dominio Active Directory è destinata a causa di centralizzata funzionalità di gestione accesso e la configurazione che offre non solo ai sistemi Windows, ma agli altri client. Qualsiasi altra directory o applicazione che fornisce l'autenticazione e la configurazione di servizi di gestione può e verranno destinate dai pirati informatici determinati. Anche se questo documento è incentrato sulla protezione che consente di ridurre la probabilità di una compromissione delle installazioni Active Directory, tutte le organizzazioni che includa i computer non Windows, archivi directory, le applicazioni o dati devono preparare anche per gli attacchi contro questi sistemi.  
  
## <a name="initial-breach-targets"></a>Destinazioni di violazione iniziale  
Nessuno intenzionalmente crea un'infrastruttura IT che espone l'organizzazione per compromettere. Quando viene costruito un insieme di strutture Active Directory, è in genere invariati e corrente. Come passare anni e vengono acquisite nuovi sistemi operativi e applicazioni, hai aggiunto all'insieme di strutture. Come vengono riconosciuti i vantaggi di gestibilità che fornisce Active Directory più contenuto viene aggiunto alla directory, più persone integrano i propri computer o applicazioni di dominio Active Directory e i domini vengono aggiornati per supportare la nuova funzionalità offerta dalle versioni più recenti del sistema operativo Windows. Cosa accade anche nel tempo, tuttavia, è che anche se è una nuova infrastruttura da aggiungere altre parti dell'infrastruttura potrebbero non essere mantenuti anche come erano inizialmente, sistemi e applicazioni funzionino correttamente e pertanto non ricevono attenzione e le organizzazioni iniziano dimenticarsi che essi non eliminato la propria infrastruttura legacy. In base a quello che possiamo osservare la valutazione di infrastrutture di compromesse, meno recenti, più grandi e complesse l'ambiente e più probabile è che esistono numerose istanze delle vulnerabilità sfruttata comunemente.  
  
Indipendentemente dalla motivazione dell'utente malintenzionato, la maggior parte delle violazioni della sicurezza di informazioni iniziare con la compromissione di uno o due sistemi alla volta. Questi eventi iniziali o punti di ingresso alla rete, spesso sfruttano vulnerabilità che avrebbero potuto essere riparate, ma non sono stati. Il [2012 dati violazione indagini Report (DBIR)](http://www.verizonbusiness.com/resources/reports/rp_data-breach-investigations-report-2012_en_xg.pdf), che è un study annuale generati dal Team del rischio Verizon in collaborazione con un numero di agenzie di sicurezza e altre aziende, gli Stati che erano "non estremamente difficile" 96 percento degli attacchi, e che "97% delle violazioni sono evitabili tramite controlli semplici o intermedi." Questi risultati possono essere una conseguenza diretta delle vulnerabilità sfruttata comunemente che seguono.  
  
### <a name="gaps-in-antivirus-and-antimalware-deployments"></a>Eventuali lacune nei programmi Antivirus e Antimalware distribuzioni  
*Legge numero otto: uno scanner di malware non aggiornati è praticamente all'alcun programma antivirus solo affatto.* - [Ten Immutable Laws of Security (Version 2.0)](https://technet.microsoft.com/security/hh278941.aspx)  
  
Analisi delle distribuzioni di antivirus e antimalware delle organizzazioni rivela spesso un ambiente in cui la maggior parte delle workstation sono configurati con software antivirus e antimalware che è abilitato e corrente. Eccezioni sono in genere le workstation che si connettono raramente per i dispositivi aziendali di ambiente o dipendente per quali programmi antivirus e antimalware può essere difficili da distribuire, configurare e aggiornare software.  
  
La popolazione di server, tuttavia, tendono a essere che meno protetto in modo coerente in molti ambienti di compromessi. Come indicato nel [2012 dati violazione indagini](http://www.verizonbusiness.com/resources/reports/rp_data-breach-investigations-report-2012_en_xg.pdf)94% di tutti i dati compromette i server coinvolti, che rappresenta un incremento del 18% nel corso dell'anno precedente e incorporati di 69% di attacchi malware. In popolazioni di server, non è insolito per trovare che le installazioni di antivirus e antimalware sono configurate in modo incoerente, aggiornate, non configurato correttamente o anche disabilitate. In alcuni casi, il software antivirus e antimalware è disabilitato per il personale amministrativo, ma in altri casi, gli utenti malintenzionati disattivare il software dopo compromettere un server tramite altri problemi di protezione. Quando il software antivirus e antimalware è disabilitato, gli utenti malintenzionati quindi prevede malware sul server e lo stato attivo sulla propagazione compromesso tra la popolazione di server.  
  
È importante non solo per garantire che i sistemi siano protetti con corrente e completa di malware protection, ma anche per monitorare i sistemi per la disabilitazione o la rimozione del software antivirus e antimalware e il riavvio automatico di protezione quando è disattivato manualmente. Anche se nessun software antivirus e antimalware in grado di garantire la prevenzione e il rilevamento di infezioni tutti, un'implementazione correttamente configurata e distribuita antivirus e antimalware può ridurre la probabilità di infezione.  
  
### <a name="incomplete-patching"></a>L'applicazione di patch incompleto  
*Legge numero tre: se non tenere il passo con correzioni per la sicurezza, la rete non sarà tuo per molto tempo.* - [10 Immutable Laws of Security Administration](https://technet.microsoft.com/library/cc722488.aspx)  
  
Microsoft rilascia bollettini sulla sicurezza il secondo martedì di ogni mese, anche se in rari casi gli aggiornamenti della sicurezza rilasciati tra gli aggiornamenti della sicurezza mensile (si tratta di noto anche come "out-of-band" aggiornamenti) quando la vulnerabilità viene considerata un rischio urgente ai sistemi dei clienti. Se una piccola azienda consente di configurare i computer Windows per utilizzare Windows Update per la gestione del sistema e l'applicazione di patch applicazione o una grande organizzazione Usa software di gestione, ad esempio System Center Configuration Manager (SCCM) per distribuire le patch in base a piani dettagliate, gerarchici, molti clienti patch le proprie infrastrutture di Windows in modo tempestivo relativamente.  
  
Tuttavia, alcuni infrastrutture includono solo i computer Windows e applicazioni Microsoft e negli ambienti di compromessi, è comune per trovare che strategia di gestione delle patch dell'organizzazione contiene spazi. I sistemi in questi ambienti Windows in modo incoerente sono corretti. Sistemi operativi Windows non vengono corretti in alcuni casi, se affatto. Applicazioni di (COTS) commerciali preconfigurate contengono vulnerabilità per cui sono disponibili patch, ma non sono state applicate. I dispositivi di rete sono spesso configurati con le credenziali predefinite di fabbrica e non firmware Aggiorna anni dopo la loro installazione. Le applicazioni e sistemi operativi che non sono più supportati dai rispettivi produttori vengono spesso archiviati in esecuzione, nonostante non può essere corretto contro vulnerabilità. Ciascuno di questi sistemi senza patch rappresenta un altro punto di ingresso potenziali per gli utenti malintenzionati.  
  
L'informatica di consumo IT ha introdotte ulteriori sfide che il dipendente proprietà dispositivi vengono usati per accedere ai dati aziendali proprietà e l'organizzazione non potrebbe avere a nessun controllare l'applicazione di patch e la configurazione dei dispositivi personali dei dipendenti. Hardware di livello aziendale in genere viene fornito con le opzioni di configurazione aziendale e le funzionalità di gestione, ma con meno scelta nella selezione singoli dispositivi e di personalizzazione. Dipendente incentrate sull'hardware offre una gamma più ampia di produttori, fornitori, funzionalità di sicurezza hardware, le funzionalità di protezione software, le funzionalità di gestione e opzioni di configurazione e molte funzionalità enterprise può essere completamente assente.  
  
#### <a name="patch-and-vulnerability-management-software"></a>Software di gestione delle vulnerabilità e delle patch  
Se un sistema di gestione delle patch efficace è attiva per i sistemi Windows e le applicazioni Microsoft, parte della superficie di attacco che creano vulnerabilità senza patch è stato risolto. Tuttavia, a meno che i sistemi non Windows, le applicazioni non Microsoft, infrastruttura di rete e dispositivi dei dipendenti siano mantenuti aggiornati sulle patch e altre correzioni, l'infrastruttura rimane vulnerabile. In alcuni casi, il fornitore dell'applicazione può offrire funzionalità di aggiornamento automatico; in altre, potrebbe essere necessario per elaborare un approccio per recuperare e applicare patch e altre correzioni regolarmente.
  
### <a name="outdated-applications-and-operating-systems"></a>Sistemi operativi e applicazioni non aggiornate  
*"Non è previsto un sistema operativo di sei anni per la protezione da attacchi di sei mese precedente".* -Informazioni protezione Professional con 10 anni di esperienza le installazioni aziendali di protezione  
  
Anche se "ottenere corrente, Rimani aggiornato" potrebbe sembrare una frase marketing, applicazioni e sistemi operativi obsoleti creano rischi in molte organizzazioni le infrastrutture IT. Un sistema operativo che è stato rilasciato nel 2003 potrebbe ancora essere supportato dal fornitore e fornito con gli aggiornamenti per le vulnerabilità di indirizzo, ma tale sistema operativo non può contenere le funzionalità di sicurezza aggiunte nelle versioni più recenti del sistema operativo. Sistemi non aggiornati possono anche richiedere indebolire determinati servizi di dominio Active Directory configurazione della protezione per supportare le funzionalità di minore di tali computer.  
  
Le applicazioni che sono stati scritti per utilizzare i protocolli di autenticazione legacy dai fornitori che non supportano in genere l'applicazione non possono essere retooled per supportare i meccanismi di autenticazione più avanzati. Tuttavia, il dominio di Active Directory di un'organizzazione potrebbe ancora configurato per l'archiviazione di hash di LAN Manager o crittografate per supportare tali applicazioni. Le applicazioni scritte prima l'introduzione dei sistemi operativi più recenti potrebbero non funzionare bene affatto in corrente i sistemi operativi, le organizzazioni devono gestire sistemi meno recenti e meno recenti e in alcuni casi, non completamente supportato hardware e software.  
  
Anche nei casi in cui le organizzazioni hanno aggiornato i controller di dominio di Windows Server 2012, Windows Server 2008 R2 o Windows Server 2008, è comune per trovare porzioni significative del membro compilazione server sia in esecuzione Windows Server 2003, ovvero non è più nella fase del supporto mainstream, o anche Windows 2000 Server o Windows NT Server 4.0 (che non sono completamente supportati). Più un'organizzazione gestisce i sistemi di durata più aumenta la differenza tra insiemi di funzionalità e più è probabile che diventa sistemi di produzione saranno supportati. Inoltre, il tempo viene mantenuta una foresta Active Directory, più notare che le applicazioni e sistemi legacy non sono installate nei piani di aggiornamento. Questo può significare che un singolo computer che eseguono una singola applicazione può introdurre vulnerabilità del dominio o foresta poiché Active Directory è configurato per supportare i protocolli legacy e meccanismi di autenticazione.  
  
Per eliminare le applicazioni e sistemi legacy, è consigliabile concentrarsi sull'identificazione e la loro catalogazione e quindi su come determinare se aggiornare o sostituire l'applicazione o un host. Anche se può essere difficile trovare le sostituzioni per le applicazioni altamente specializzate per cui esiste supporto né un percorso di aggiornamento, potrebbe essere in grado di sfruttare il concetto di "distruzione creativa" per sostituire l'applicazione legacy con una nuova applicazione che fornisce le funzionalità necessarie. [Pianificazione del compromesso](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Planning-for-Compromise.md) è descritta in modo più approfondito nella "Pianificazione per la compromissione" più avanti in questo documento.  
  
### <a name="misconfiguration"></a>Errore di configurazione  
*Legge numero quattro: non è una buona soluzione per installare correzioni per la sicurezza in un computer protetto è stato mai per iniziare.* - [10 Immutable Laws of Security Administration](https://technet.microsoft.com/library/cc722488.aspx)  
  
Anche in ambienti in cui vengono mantenuti in genere sistemi e le patch correnti, in genere rendiamo spazi vuoti o errori di configurazione del sistema operativo, applicazioni in esecuzione su computer e Active Directory. Alcuni errori di configurazione espongono solo il computer locale per compromettere, ma dopo che un computer è "proprietario", gli utenti malintenzionati in genere lo stato attivo sulla propagazione ulteriormente il compromesso tra altri sistemi e infine in Active Directory. Di seguito sono alcune delle aree comuni in cui è identificare le configurazioni che introducono rischi.  
  
#### <a name="in-active-directory"></a>In Active Directory  
Gli account in Active Directory che in genere sono destinati dai pirati informatici sono quelli che sono membri dei gruppi con privilegi più elevati, ad esempio i membri dei gruppi Administrators (BA) incorporati in Active Directory, Domain Admins (DA) o Enterprise Admins (EA). L'appartenenza di questi gruppi deve essere ridotto il minor numero di account possibili, in modo da limitare la superficie di attacco di questi gruppi. È anche possibile eliminare questi gruppi con privilegi, l'appartenenza "permanente" vale a dire, è possibile implementare le impostazioni che consentono di temporaneamente popolare questi gruppi solo quando sono necessari i privilegi a livello di dominio e di foresta. Quando si utilizzano account con privilegi elevati, devono essere utilizzati solo nei sistemi protetti designati come controller di dominio o degli host amministrativi protetti. Vengono fornite informazioni dettagliate per implementare tutte queste configurazioni [riduzione della superficie di attacco Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md).  
  
Quando si valuta l'appartenenza di gruppi con privilegi più elevati in Active Directory, troviamo comunemente appartenenza eccessivo in tutti e tre i gruppi con privilegi più. In alcuni casi, le organizzazioni hanno decine, anche centinaia di account nei gruppi DA. In altri casi, le organizzazioni inserire gli account direttamente in gruppo incorporato Administrators, pensare che il gruppo è "meno privilegiato" rispetto al gruppo DAs. Non è. È spesso possibile trovare un numero limitato di membri permanenti del gruppo EA nel dominio radice della foresta, nonostante il fatto che EA sono raramente e temporaneamente richiesti i privilegi. Trovare un IT quotidiane amministrativi account utente in tutti e tre i gruppi è anche comune, anche se si tratta di una configurazione in modo efficace ridondante. Come descritto in [riduzione della superficie di attacco Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md), se un account è membro permanente di uno di questi gruppi o tutti gli elementi, l'account è utilizzabile per compromettere e persino distruggere l'ambiente di dominio Active Directory e i sistemi e gli account di gestione. Vengono forniti suggerimenti per la configurazione protetta e l'utilizzo di account con privilegi in Active Directory in [riduzione della superficie di attacco Active Directory](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Reducing-the-Active-Directory-Attack-Surface.md).  
  
#### <a name="on-domain-controllers"></a>Nei controller di dominio  
Quando abbiamo valutare i controller di dominio, troviamo spesso trovarli configurate e gestite in modo diverso rispetto ai server membro. Controller di dominio eseguono a volte lo stesso applicazioni e utilità installate sui server membri, dato che non sono più necessarie nei controller di dominio, ma poiché le applicazioni fanno parte di una build standard. Queste applicazioni possono fornire funzionalità minime nei controller di dominio ma aggiungere in modo significativo la superficie di attacco richiedendo l'impostazione di configurazione che aprire le porte, creare gli account del servizio con privilegi elevati o concedere l'accesso al sistema dagli utenti che non deve connettersi a un controller di dominio per scopi diversi da autenticazione e l'applicazione di criteri di gruppo. In alcuni violazioni, gli utenti malintenzionati sono utilizzati strumenti che sono stati già installati nei controller di dominio non solo per ottenere l'accesso ai controller di dominio, ma per modificare o danneggiare il database di Active Directory.  
  
Quando si estrae le impostazioni di configurazione di Internet Explorer nei controller di dominio, riteniamo che gli utenti accedono con gli account che dispongono di livelli elevati di privilegi in Active Directory e sono utilizzati gli account per accedere a Internet e intranet dai controller di dominio. In alcuni casi, gli account di aver configurato le impostazioni di Internet Explorer nei controller di dominio per consentire il download di Internet al contenuto e utilità freeware sono state scaricate da siti Internet e installata nei controller di dominio. Sicurezza avanzata di Internet Explorer è abilitata per utenti e amministratori per impostazione predefinita, ma spesso osservare che è stata disabilitata per gli amministratori. Quando un account con privilegiato elevati accede a Internet e scarica il contenuto a qualsiasi computer, tale computer viene messa a rischio grave. Quando il computer è un controller di dominio, l'intera installazione di servizi di dominio Active Directory viene messa a rischio.  
  
##### <a name="protecting-domain-controllers"></a>Proteggere i controller di dominio  
I controller di dominio devono essere considerati componenti critici dell'infrastruttura, protetti più rigorosamente e configurati in modo più rigido rispetto a file, stampa e server applicazioni. I controller di dominio non è necessario a eseguire qualsiasi software che non è necessario per il controller di dominio alla funzione o non protegge il controller di dominio dagli attacchi. I controller di dominio devono essere consentiti l'accesso a Internet e le impostazioni di sicurezza devono essere configurate e applicate dagli oggetti Criteri di gruppo (GPO). Indicazioni dettagliate per l'installazione sicura, configurazione e la gestione dei controller di dominio sono incluse nei [protezione controller di dominio dagli attacchi](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md).  
  
#### <a name="within-the-operating-system"></a>All'interno del sistema operativo  
*Legge numero due: se un utente malintenzionato può modificare il sistema operativo nel computer, non è il computer più.* - [Ten Immutable Laws of Security (Version 2.0)](https://technet.microsoft.com/security/hh278941.aspx)  
  
Anche se alcune organizzazioni creare configurazioni di base per i server di diversi tipi e consentono limitato la personalizzazione del sistema operativo dopo l'installazione, l'analisi di ambienti di compromessi individua spesso un numero elevato di server in modo ad hoc distribuito e configurato manualmente e in modo indipendente. Configurazioni tra due server che esegue la stessa funzione possono essere completamente diverse, in cui nessun server è configurato in modo sicuro. Al contrario, linee di base di configurazione server possono essere applicate in modo coerente, ma anche in modo coerente non configurato correttamente; ovvero, i server sono configurati in modo che crea la stessa vulnerabilità in tutti i server di un determinato tipo. Errore di configurazione include procedure consigliate, ad esempio la disattivazione della funzionalità di sicurezza, concessione eccessivo diritti e autorizzazioni per gli account (in particolare gli account di servizio), utilizzare credenziali locali identiche diversi sistemi e che consenta di installazione di applicazioni non autorizzate e le utilità creare vulnerabilità i propri.  
  
##### <a name="disabling-security-features"></a>La disattivazione delle funzionalità di sicurezza  
Le organizzazioni a volte disattivare Windows Firewall con protezione avanzata (WFAS) dalla convinzione che WFAS è difficile da configurare o richiede la configurazione di lavoro intensi sul lato. Tuttavia, a partire da Windows Server 2008, quando è installato qualsiasi ruolo o funzionalità in un server, è configurato per impostazione predefinita con i privilegi minimi necessari per il ruolo o funzionalità a funzione e Windows Firewall viene configurato automaticamente per supportare il ruolo o funzionalità. Disabilitando WFAS (e non si utilizza un altro firewall basato su host al suo posto), le organizzazioni aumentano la superficie di attacco dell'intero ambiente Windows. Perimeter firewalls provide some protection against attacks that directly target an environment from the Internet, but they provide no protection against attacks that exploit other attack vectors such as [drive-by download](https://www.microsoft.com/security/sir/glossary/drive-by-download-sites.aspx) attacks, or attacks that originate from other compromised systems on the intranet.  
  
Le impostazioni di controllo dell'account utente a volte sono disattivate in server perché il personale amministrativo trovare le istruzioni intrusivo. Although [Microsoft Support article 2526083](https://support.microsoft.com/kb/2526083) describes scenarios in which UAC may be disabled on Windows Server, unless you are running a server core installation (where UAC is disabled by design), you should not disable UAC on servers without careful consideration and research.  
  
In altri casi, le impostazioni del server sono configurate per meno sicuri valori poiché le organizzazioni si applicano le impostazioni di configurazione server non aggiornato a nuovi sistemi operativi, ad esempio l'applicazione delle linee di base di Windows Server 2003 a computer che eseguono Windows Server 2012, Windows Server 2008 R2 o Windows Server 2008, senza modificare le linee di base per riflettere le modifiche nel sistema operativo. Invece di esecuzione linee di base server precedente a nuovi sistemi operativi, quando si distribuisce un nuovo sistema operativo, esaminare le modifiche di sicurezza e le impostazioni di configurazione per assicurarsi che le impostazioni implementate sono applicabili e appropriato per il nuovo sistema operativo.  
  
##### <a name="granting-excessive-privilege"></a>Concedere privilegi eccessivi  
In quasi ogni ambiente verranno, viene concesso privilegi eccessivi agli account locali e basati su dominio nei sistemi Windows. Gli utenti vengono concessi diritti di amministratore locale sulle proprie workstation, server membri di eseguire i servizi configurati con diritti oltre necessarie per funzionare e gruppi di amministratori locali la popolazione server contengono decine o addirittura centinaia di locali e gli account di dominio. Compromissione di un solo account con privilegi in un computer consente a utenti malintenzionati compromettere gli account di ciascun utente e un servizio che accede al computer e raccolto e permette di sfruttare le credenziali per propagare la compromissione ad altri sistemi.  
  
Anche se pass-the-hash (PTH) e altri attacchi contro il furto di credenziali vengono generati oggi, che perché è disponibili gratuitamente strumenti che rende più semplice e facile da estrarre le credenziali di altri account con privilegi quando un utente malintenzionato riesca ad accedere a livello di amministratore o di sistema a un computer. Anche senza strumenti che consente la raccolta delle credenziali da sessioni di accesso, un utente malintenzionato con privilegi di accesso a un computer può installare facilmente keystroke logger che acquisire le pressioni di tasti, screenshot e il contenuto degli Appunti. An attacker with privileged access to a computer can disable antimalware software, install rootkits, modify protected files, or install malware on the computer that automates attacks or turns a server into a [drive-by download](https://www.microsoft.com/security/sir/glossary/drive-by-download-sites.aspx) host.  
  
Variano tattiche utilizzate per estendere una violazione oltre a un singolo computer, ma la chiave per la propagazione di compromissione è l'acquisizione di accesso con privilegiato elevati per altri sistemi. Riducendo il numero di account con privilegi di accesso a qualsiasi sistema, ridurre la superficie di attacco non solo di quel computer, ma la probabilità di un utente malintenzionato raccolta delle credenziali importanti dal computer.  
  
##### <a name="standardizing-local-administrator-credentials"></a>La standardizzazione delle credenziali di amministratore locale  
Non esiste già da tempo discussioni tra gli addetti alla sicurezza come a se non esiste valore rinominare l'account amministratore locale sul computer Windows. Che cos'è effettivamente importante sugli account di amministratore locale è fatto che siano configurati con lo stesso nome utente e password tra più computer.  
  
Se l'account Administrator locale è denominato sullo stesso valore tra server e la password assegnata all'account è configurata con lo stesso valore, gli utenti malintenzionati possono estrarre le credenziali dell'account in un computer in cui amministratore o a livello di sistema è stato ottenuto l'accesso. L'autore dell'attacco non dispone di inizialmente compromettere l'account amministratore. è necessario solo compromettere l'account di un utente che è un membro del gruppo Administrators locale o di un account di servizio che è configurato per l'esecuzione come LocalSystem o con privilegi di amministratore. L'autore dell'attacco può quindi estrarre le credenziali per l'account amministratore e riprodurre tali credenziali negli accessi alla rete in altri computer nella rete.  
  
Come un altro computer ha un account locale con il medesimo nome utente e password (o hash password) come le credenziali dell'account che sono viene presentate, il tentativo di accesso ha esito positivo e l'autore dell'attacco ottiene accesso con privilegi al computer di destinazione. In current versions of Windows, the built-in Administrator account is [disabled by default](https://technet.microsoft.com/library/cc753450.aspx), but in legacy operating systems, the account is enabled by default.  
  
> [!NOTE]  
> Alcune organizzazioni hanno configurato intenzionalmente gli account amministratore locali deve essere abilitata nella convinzione che questo fornisca "alternativa" nel caso in cui tutti gli altri account con privilegi sono bloccati da un sistema. However, even if the local Administrator account is disabled and there are no other accounts available that can enable the account or log on to the system with Administrator privileges, the system can be booted into safe mode and the built-in local Administrator account can be re-enabled, as described in [Microsoft Support article 814777](https://support.microsoft.com/kb/814777). Inoltre, se il sistema si applica comunque correttamente oggetti Criteri di gruppo, un oggetto Criteri di gruppo può essere modificato per riattivare l'account amministratore (temporaneamente) o per aggiungere un account basato su dominio al gruppo Administrators locale, è possibile configurare i gruppi con restrizioni. Operazioni di ripristino può essere eseguite e l'account amministratore nuovamente può essere disabilitata. Per impedire in modo efficace la compromissione laterale che utilizza credenziali dell'account amministratore locale predefinite, le password e nomi utente univoci devono essere configurate per gli account amministratore locali. To deploy unique passwords for local Administrator accounts via a GPO, see [Solution for management of built-in Administrator account's password via GPO](https://technet.microsoft.com/mt227395.aspx) on technet.  
  
##### <a name="permitting-installation-of-unauthorized-applications"></a>Consentire l'accesso a installazione di applicazioni non autorizzate  
*Legge numero uno: Se un utente malintenzionato può indurre di eseguire il programma nel computer, non è esclusivamente il computer più.* - [Ten Immutable Laws of Security (Version 2.0)](https://technet.microsoft.com/security/hh278941.aspx)  
  
Se un'organizzazione distribuisce le impostazioni di base coerenti tra server, l'installazione delle applicazioni che non fanno parte del ruolo definito un server deve essere consentito. Consentendo l'installazione di software che non fa parte della funzionalità designato del server, i server sono esposti a installazione accidentale o dannosa di software che aumenta la superficie di attacco del server, introduce vulnerabilità delle applicazioni o causa instabilità del sistema.  
  
#### <a name="applications"></a>Applicazioni  
Come descritto in precedenza, le applicazioni vengono spesso installate e configurate per utilizzare gli account che dispongono di che più privilegi rispetto all'applicazione richiede effettivamente. In alcuni casi, la documentazione dell'applicazione specifica che gli account di servizio devono essere membri del gruppo Administrators locale del server o devono essere configurati per l'esecuzione nel contesto di LocalSystem. Questo è spesso non perché l'applicazione richiede tali diritti, ma poiché determinare i diritti e autorizzazioni necessità di account del servizio di un'applicazione richiede investimenti in altro tempo e fatica. Se un'applicazione non viene installato con i privilegi minimi necessari per l'applicazione e delle relative funzionalità configurata funzionare, il sistema viene esposto agli attacchi che sfruttano i privilegi dell'applicazione senza un attacco contro il sistema operativo stesso.  
  
### <a name="lack-of-secure-application-development-practices"></a>Mancanza di applicazione di protezione consigliate per lo sviluppo  
Infrastruttura esistente per supportare i carichi di lavoro aziendali. In questi carichi di lavoro vengono implementate in applicazioni personalizzate, è fondamentale per garantire che le applicazioni sviluppate usando consigliate protetta. Analisi della causa principale dei problemi aziendali spesso rivela che un compromesso iniziale viene effettuato tramite personalizzata applicazioni soprattutto quelle Internet affiancate. La maggior parte di questi compromessi vengono eseguita tramite compromissione di attacchi noti, ad esempio SQL injection (SQLi) e attacchi cross-site scripting (CSS).  
  
SQL Injection è una vulnerabilità di applicazione che consente l'input definita dall'utente modificare un'istruzione SQL che viene passata al database per l'esecuzione. Questo input può essere fornito attraverso un campo di applicazione, un parametro (ad esempio, la stringa di query o un cookie) o altri metodi. Il risultato di questo tipo di inserimento è che l'istruzione SQL fornito per il database è fondamentalmente diverso rispetto a quale lo sviluppatore deve. Eseguire, ad esempio, una query comune usata nella valutazione di una combinazione nome utente/password:  
  
`SELECT userID FROM users WHERE username = 'sUserName' AND password = 'sPassword'`  
  
Quando questo viene ricevuto dal server di database, indica al server di esaminare la tabella di utenti e restituire i record di ID utente in cui il nome utente e password corrispondano a quelle fornite dall'utente (presumibilmente tramite un modulo di accesso di qualche genere). Naturalmente lo scopo dello sviluppatore in questo caso è per restituire solo un record valido se un nome utente corretto e la password può essere forniti dall'utente. Se uno è corretto, il server di database in grado di trovare un record corrispondente e restituire un risultato vuoto.  
  
Il problema si verifica quando un utente malintenzionato effettua un evento imprevisto, ad esempio per offrire i propri SQL al posto di dati validi. Poiché SQL è interpretato in volo dal server di database, il codice inserito sarebbe elaborato come se lo sviluppatore ha inserito in se stesso. Ad esempio, se l'autore dell'attacco immesso **amministratore** per l'ID utente e **xyz** o **1 = 1** sarebbe come password, l'istruzione risultante elaborato dal database:  
  
`SELECT userID FROM users WHERE username = 'administrator' AND password = 'xyz' OR 1=1`  
  
Quando la query viene elaborata dal server di database, tutte le righe nella tabella verranno restituite nella query perché 1 = 1 è sempre uguale a True, pertanto che non è importante se il nome utente corretto e la password è noto o fornito. Il risultato nella maggior parte dei casi è che l'utente verrà connesso come il primo utente nel database dell'utente. Nella maggior parte dei casi, questo sarà l'utente amministratore.  
  
Oltre a semplicemente l'accesso, istruzioni SQL non valido, ad esempio può essere utilizzato per aggiungere, eliminare, modificare i dati o anche rimuovere tabelle intere (eliminare) da un database. Nella peggiore dei casi in cui SQLi viene combinato con privilegi eccessivi, è possono eseguire i comandi del sistema operativo per abilitare la creazione di nuovi utenti, per scaricare strumenti di attacco, o per eseguire altre operazioni scelto.  
  
Il cross-site scripting, la vulnerabilità è stato introdotto nell'output dell'applicazione. Un attacco inizia con un utente malintenzionato che fornisce dati non validi all'applicazione, ma in questo caso i dati non validi sono sotto forma di codice di script (ad esempio JavaScript) che verrà eseguita da browser della vittima. Exploit di vulnerabilità di XSS un utente malintenzionato di eseguire le funzioni dell'applicazione di destinazione nel contesto dell'utente che ha avviato il browser. Gli attacchi XSS sono in genere avviati da un messaggio di posta elettronica phishing incoraggiare all'utente di selezionare un collegamento che si connette all'applicazione e viene eseguito il codice di attacco.  
  
XSS è spesso sfruttato negli scenari di e-commerce in cui un utente malintenzionato può effettuare acquisti o trasferire denaro nel contesto dell'utente interessato e i servizi bancari in linea. Nel caso di un attacco di destinazione in un'applicazione di gestione identità personalizzata basata sul web, un utente malintenzionato di creare le proprie identità, modificare le autorizzazioni e diritti e causare una compromissione del sistema.  
  
Anche se una trattazione completa di cross-site scripting e SQL injection esula dall'ambito di questo documento, il [aprire Web applicazione sicurezza progetto (OWASP)](https://www.owasp.org/index.php/Main_Page) pubblica un elenco primi 10 con informazioni dettagliate sulle vulnerabilità e alle contromisure.  
  
Indipendentemente dall'investimento nell'infrastruttura la protezione, se progettate e scritte applicazioni distribuite all'interno dell'infrastruttura, l'ambiente vulnerabile agli attacchi Infrastrutture protette anche ben spesso non possono fornire contromisure efficaci per questi attacchi di applicazione. Il problema di composizione, le applicazioni progettate in modo non corretto potrebbero richiedere che gli account del servizio siano concesse autorizzazioni eccessivo per l'applicazione alla funzione.  
  
Microsoft Security Development Lifecycle (SDL) è un set di controlli strutturali processo che interagiscono per migliorare l'inizio di sicurezza nella fase iniziale di requisiti di raccolta e l'estensione tramite il ciclo di vita dell'applicazione fino a quando non si sono state rimosse. Questa integrazione dei controlli di sicurezza efficace non solo è critica dal punto di vista della sicurezza, è fondamentale garantire che la sicurezza delle applicazioni è costo e pianificazione attiva. Valutazione di un'applicazione per problemi di protezione quando è in realtà codice completo richiede alle organizzazioni di prendere decisioni sulla protezione dell'applicazione solo prima o dopo l'applicazione è stata distribuita. Un'organizzazione può scegliere di risolvere i difetti di applicazione prima di distribuire l'applicazione nell'ambiente di produzione, incorrere in costi e ritardi, o l'applicazione può essere distribuito nell'ambiente di produzione con difetti di protezione noti, l'organizzazione per compromettere l'esposizione.  
  
Alcune organizzazioni posizionare i costi di correggere un problema di protezione nel codice di produzione sopra $10.000 al problema e le applicazioni sviluppate senza un efficace SDL possono Media di più di dieci problemi a livello di gravità elevato per 100.000 righe di codice. Nelle applicazioni di grandi dimensioni, i costi inoltrare rapidamente. Al contrario, molte società impostare un benchmark di meno di un problema per 100.000 righe di codice in fase di esaminare il codice finale del processo SDL e puntare a zero problemi nelle applicazioni ad alto rischio nell'ambiente di produzione.  
  
Implementazione del processo SDL migliora la protezione includendo i requisiti di sicurezza all'inizio nella raccolta dei requisiti e progettazione di un'applicazione fornisce modellazione per applicazioni ad alto rischio. richiede di formazione e il monitoraggio degli sviluppatori; e richiede chiaro, codice coerente standard e procedure consigliate. L'effetto netto di un SDL è miglioramenti significativi in sicurezza delle applicazioni, riducendo i costi per sviluppare, distribuire, gestire e rimuovere le autorizzazioni di un'applicazione. Although a detailed discussion of the design and implementation of SDL is beyond the scope of this document, refer to the [Microsoft Security Development Lifecycle](https://www.microsoft.com/security/sdl/default.aspx) for detailed guidance and information.
